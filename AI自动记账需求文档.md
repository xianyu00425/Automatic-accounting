# AI 自动记账 H5 - 产品需求文档 (PRD)

> **版本**: V8/V9  &nbsp;|&nbsp; **平台**: 移动端 H5  &nbsp;|&nbsp; **技术**: 纯前端（HTML + CSS + JS + localStorage）

---

## 一、产品概述

### 1.1 产品定位

一款面向中国用户的移动端 H5 自动记账工具，支持从微信支付和支付宝导入账单文件，自动分类处理交易记录，提供费用分析、预算管理、账户管理和转账记录等完整记账功能。

### 1.2 核心设计理念

- **零后端依赖**：所有数据存储于浏览器 localStorage，无需服务器
- **导入即记账**：账单导入后自动解析、批量分类、即时出报表
- **移动端优先**：最大宽度 480px 的响应式布局，适配 iPhone 刘海屏

### 1.3 技术架构

| 层级 | 技术选型 |
|------|----------|
| 结构层 | HTML5（单页应用，4 个页面 + 18 个弹窗） |
| 样式层 | CSS3（CSS 变量、Grid、Flexbox、动画） |
| 逻辑层 | 原生 JavaScript（ES5/ES6 混合） |
| 数据层 | localStorage（6 个存储键） |
| 外部依赖 | Font Awesome 6.4（图标）、SheetJS 0.18.5（Excel 解析）、Inter 字体 |

## 二、用户角色与场景

### 2.1 目标用户

个人用户，主要使用微信支付和支付宝进行日常消费，需要：
- 定期导入支付平台的账单文件进行财务汇总
- 查看每月的支出/收入构成饼图
- 设置预算并跟踪超支情况
- 管理多个资金账户（支付宝、微信、银行卡等）
- 记录账户间的转账

### 2.2 核心使用场景

| 场景 | 描述 |
|------|------|
| 导入账单 | 用户从微信/支付宝导出账单文件，导入到对应账户 |
| 批量分类 | 系统根据预设规则自动分类，用户手动处理未匹配的交易 |
| 查看报表 | 在首页查看当月收支统计和分类饼图 |
| 管理预算 | 设置月度/年度预算，首页查看进度条 |
| 记录转账 | 记录支付宝到银行卡等账户间转账 |
| 数据备份 | 导出/导入 JSON 格式的数据备份 |

| 功能 | 通过左右箭头切换查看的月份 |
| 联动 | 影响所有模块的月份筛选：收支统计卡片、费用分析饼图、预算进度 |

#### 3.2.2 收支统计卡片

| 卡片 | 数据源 | 计算公式 | 显示颜色 |
|------|--------|----------|----------|
| 本月收入 | 当月所有 `type=income` 的交易 | `Σ |amount|` | 绿色 |
| 本月支出 | 当月所有 `type=expense` 的交易 | `Σ |amount|` | 橙红色 |
| 本月结余 | — | `收入 - 支出` | 蓝色 |
| 账户总余额 | 所有账户余额之和 | `Σ account.balance` | 紫色 |

#### 3.2.3 费用分析

| 项目 | 说明 |
|------|------|
| 展示形式 | Canvas 绘制的环形饼状图（甜甜圈图） |
| 数据来源 | **仅统计已分类的账单**（`categoryId` 存在且非空的交易） |
| 数据筛选 | 按当前选择月份 + 类型（支出/收入）+ 已分类 筛选交易 |
| 聚合逻辑 | 按分类名称汇总金额，取前 8 个分类 |
| 占比计算 | `（该分类金额 / 当月该类型已分类总金额）× 100%` |
| 交互 | 顶部"支出"/"收入"两个 Tab 切换 |
| 空状态 | 显示"暂无数据"及数据分布提示 |
| 图例 | 每个分类显示：色块、名称、金额、占比百分比 |

> **重要说明**：费用分析的数据来源于已处理了分类的账单明细。未分类的交易（`categoryId` 为空）不计入费用分析统计。用户需要先在"待分类"页面完成分类处理后，费用分析饼图才会显示对应数据。

#### 3.2.4 预算进度

| 预算类型 | 总预算（不指定分类）/ 分类预算（指定分类） |

## 六、业务流程

### 6.1 完整记账流程

```
用户从微信/支付宝导出账单文件
         ↓
    打开记账页面
         ↓
    点击"导入账单"
         ↓
    选择账户（如"支付宝"）
         ↓
    选择模板（如"支付宝"）
         ↓
    选择文件上传 / 粘贴文本
         ↓
    系统解析文件，展示预览
    （成功X条 / 重复Y条 / 跳过Z条）
         ↓
    确认导入
         ↓
    系统保存交易 + 重算余额 + 刷新视图
         ↓
    首页显示最新统计数据
         ↓
    如有未分类交易 → 待分类页面显示提醒
         ↓
    用户逐条处理 或 点击"批量处理"
         ↓
    费用分析饼图实时更新
```

### 6.2 账户余额计算流程

```
初始化时从交易历史重算：
  账户余额 = 初始余额
           + Σ(该账户的收入金额)
           - Σ(该账户的支出金额)
           - Σ(该账户作为转出方的转账金额)
           + Σ(该账户作为转入方的转账金额)

每次交易变更时增量更新：
  新增交易 → 对应账户增减余额
  编辑交易 → 先回滚旧余额，再应用新余额
  删除交易 → 回滚对应账户余额
  转账操作 → 同时更新转出方和转入方余额
```

### 6.3 费用分析数据流程

```
用户切换月份或切换支出/收入Tab
         ↓
读取 AppState.currentMonth（当前选择的月份）
         ↓
aggregateByCategory(type) 筛选交易：
  1. 获取所有交易 getTransactions()
  2. 过滤条件：
     a. t.type === type（支出/收入）
     b. date.getFullYear() === targetYear
     c. date.getMonth() === targetMonth
     d. t.categoryId 存在且非空（已分类）
         ↓
按分类名称汇总金额，计算占比
         ↓
Canvas 绘制环形饼状图 + 图例列表
```

## 八、非功能需求

### 8.1 性能要求

| 指标 | 要求 |
|------|------|
| 首屏加载 | 3 秒内显示主界面（含启动页超时保护） |
| 导入解析 | 1000 条交易在 5 秒内完成 |
| 页面切换 | 无明显卡顿（0.3s 动画过渡） |
| 饼图渲染 | Canvas 绘制，50ms 内完成 |

### 8.2 兼容性

| 项目 | 要求 |
|------|------|
| 浏览器 | Chrome 90+、Safari 14+、微信内置浏览器 |
| 设备 | iPhone 6 及以上、主流 Android 手机 |
| 屏幕 | 最小宽度 320px，适配刘海屏（safe-area-inset） |
| 文件格式 | .csv、.xlsx、.xls |

### 8.3 数据安全

- 所有数据存储在本地浏览器，不上传任何服务器
- 用户可通过导出功能备份 JSON 文件
- 清空操作需二次确认
